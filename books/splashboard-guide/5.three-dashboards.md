splashboard には dashboard が 3 種類あります。 home 、 project 、 per-repo 。

呼ばれる場面と狙いが違うだけで、 TOML の書き方自体は共通です。この章ではどれがいつ呼ばれるかと、どう使い分けるかだけを押さえます。 per-repo の安全性まわり ( trust モデル ) の話は Part 3 で深掘りします。

## どれがいつ呼ばれるか

シェル起動と `cd` のたびに、 splashboard は今いる場所を見てどの dashboard を出すかを決めます。判定は次の順序で行われます。

1. いまのディレクトリが `$HOME` 自体か
    - Yes → `home.dashboard.toml` を出す ( **home** )
    - No → 次へ
2. いまのディレクトリに `./.splashboard/dashboard.toml` ( または `./.splashboard.toml` ) があるか
    - Yes → そのファイルを出す ( **per-repo** )
    - No → 次へ
3. いまのディレクトリが git レポジトリのルートか
    - Yes → `$HOME/.splashboard/project.dashboard.toml` を出す ( **project** )
    - No → 次へ
4. どれにも当てはまらない場合
    - シェル起動時 → `home.dashboard.toml` を出す ( フォールバック )
    - `cd` のとき → 何も再描画しない ( 前の splash が画面に残る )

押さえておくと混乱しないポイントが 3 つあります。

- **`cd` では非 git ディレクトリへ降りても home は出ない**。 `cd /tmp` や `cd ~/Documents` では再描画されず、前の splash がそのまま画面に残ります
- **per-repo はそのディレクトリ自身に `.splashboard/` がある場合のみ発火**。 `cd your-repo/src/` のようにサブディレクトリへ降りた場合、 `cd` では何も再描画されません ( サブディレクトリに入るたびに重い splash が再描画される事故を防ぐためです )
- **per-repo の `.splashboard/` がなくても、 git レポジトリのルートでは project dashboard が出る**。各レポにわざわざ設定を置かなくても、どの git レポでも共通のレポ用 splash が手に入ります

## それぞれの狙い

### Home dashboard

- 場所: `$HOME/.splashboard/home.dashboard.toml`
- 出るタイミング: 新しいシェルを開いたとき、または `cd $HOME` で $HOME 自体に降りたとき
- 向いている内容: アンビエント情報 ( 天気、月相、年の進捗、 contribution heatmap 、 RSS フィード、 Hacker News のトップなど )

シェルを開いたときに眺めて気分が上がる系の情報を置く場所です。すぐアクションを促す情報 ( この PR 見て、この CI 直して ) を置きすぎると、シェルを開くたびに作業モードに引っ張られるので、 home はもう少し受動的なものに寄せた方が長持ちします。

### Project dashboard

- 場所: `$HOME/.splashboard/project.dashboard.toml`
- 出るタイミング: 自前の `.splashboard/` を持たない git レポ root に `cd` したとき
- 向いている内容: どのレポでも知りたい共通の現状 ( branch / CI / 待ち PR / 最近の commit / 最新タグ )

どのレポでも同じ視点で見たい情報を置きます。 GitHub 連携や git status を中心に組むのが基本構成です。 `project_github` preset がそのままの雛形になるので、最初はそこから削っていくのが手早いです。

### Per-repo dashboard

- 場所: レポジトリ直下の `.splashboard/dashboard.toml` ( または `.splashboard.toml` 単一ファイル )
- 出るタイミング: そのレポジトリのルートに `cd` したとき
- 向いている内容: そのレポ固有のコンテキスト ( 専用の onboarding 情報、運用 dashboard 、 README には書きにくい現状 )

per-repo は splashboard の中心機能のひとつです。レポジトリに `.splashboard/dashboard.toml` をコミットして同梱すると、 `git clone` して `cd` したその瞬間に、そのレポ専用の splash が出ます。

具体的な用途:

- OSS の onboarding: 新規 contributor 向けにまず読むべき issue 、ビルドコマンド、最近の release など
- 運用レポジトリ: 障害対応用 runbook 、当番表、最近のアラート
- モノレポの目印: `cd monorepo/` で全体ダッシュボード、各サブパッケージ root では別々の splash ( これは将来の機能、今は最上位 root のみ反応 )

ただし、外部から取ってきた `.splashboard/dashboard.toml` のネットワーク系 widget は、明示的に許可するまで動かないようになっています。クローンしたレポの設定が勝手に外部 URL を叩く事故を防ぐためです。仕組みは Part 3 の trust 章で詳しく扱います。今はクローンしたレポの splash が `🔒 requires trust` と出たら、 `splashboard trust` を 1 回叩くと覚えておけば十分です。

## どれを最初に書くか

慣れるまでは、

1. まず `splashboard install` が書いてくれた home / project を使う
2. home に見たい情報を足し引きする
3. 自分が頻繁に触る OSS / 仕事のレポに `.splashboard/dashboard.toml` を 1 つ作って per-repo を体感する

の順で十分です。

次の章から、 dashboard.toml そのものの書き方に入ります。 widget / row / child という 3 つの要素だけで splashboard のレイアウトはすべて表現できるので、まずそこを掴みます。
