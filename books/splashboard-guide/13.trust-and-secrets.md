per-repo dashboard は便利ですが、よく考えると怪しい仕組みでもあります。

```bash
$ git clone https://github.com/somebody/something
$ cd something
# ここでいきなり外部 URL を叩く widget が走ったら ?
```

`cd` した瞬間に、見ず知らずの誰かの設定がローカルで動く可能性があります。これに対処するために splashboard は fetcher 単位の trust モデルを持っています。

この章はその設計を掘ります。 OSS に `.splashboard/` を同梱したい人や、 dotfiles を公開している人にも関係する話です。

## Safety クラス

fetcher はコンパイル時に、安全性クラスを宣言します。

| クラス | 例 | untrusted な per-repo 設定から動くか |
| --- | --- | --- |
| **Safe** | `clock` 、 `git_*` 、 `system_*` 、 `github_*` ( ホストは hardcode 、 token は `api.github.com` 以外には絶対に出ない ) | はい、常に動く |
| **Network** | RSS / カレンダーフィード / 任意 URL を `[widget.options]` で渡せるもの | `splashboard trust` を通すまで `🔒 requires trust` プレースホルダ |

分類は fetcher 単位で、ファイル単位ではありません。 1 つの per-repo dashboard に Safe な `git_status` と Network な `rss` が混在していても、 `git_status` は即時に描かれ、 `rss` だけが trust 待ちで保留される、という挙動になります。

ローカルで完結する情報は最初から見え、外を叩く部分だけ確認が要る、という分け方です。

### 固定ホストとは何か

`github_*` が Safe な理由を補足しておきます。これらの fetcher は

- 接続先 URL ( `api.github.com` ) をコードで hardcode している
- 設定ファイルから接続先を上書きできない
- 認証情報 ( `GH_TOKEN` ) は、その固定ホスト以外には出ない

ので、 untrusted な per-repo 設定がどんな options を指定しようと、 GitHub 以外には通信しないことが保証されます。

逆に、もし `http_fetch` のように URL を `[widget.options]` から指定する fetcher を作ったとしたら、認証情報がなくても Network に分類されます。設定が通信先を制御できるかどうかが分類軸です。

## どの設定ファイルが trust ゲートの対象か

trust が要るのは per-repo dashboard のみです。

| ファイル | 信頼の扱い |
| --- | --- |
| `$HOME/.splashboard/settings.toml` | 暗黙信頼 |
| `$HOME/.splashboard/home.dashboard.toml` | 暗黙信頼 |
| `$HOME/.splashboard/project.dashboard.toml` | 暗黙信頼 |
| `./.splashboard/dashboard.toml` ( or `./.splashboard.toml` ) | trust-gated |
| 何もファイルがない場合の組み込みデフォルト | 暗黙信頼 |

HOME 配下のファイルは自分で置いたものが前提なので、別途確認は要りません。 trust の対象は、クローンしてきた他人のレポに同梱されている dashboard だけです。

## 信頼の流れ

レポジトリ内で 1 行打つだけです。

```bash
# レポ内のどこからでも、近い local dashboard を遡って探す
splashboard trust

# 明示的にパス指定
splashboard trust ./path/to/.splashboard.toml

# 取り消し
splashboard revoke

# 今信頼しているもの一覧
splashboard list-trusted
```

`splashboard trust` を実行すると、 capability diff ( この config がどの Network widget を動かそうとしているか ) が表示され、 y/N で確認が走ります。

信頼情報は

```
$HOME/.splashboard/trust.toml
```

に `( canonical_path, sha256 )` のペアで保存されます。

ポイントは hash が含まれていることです。 dashboard.toml を編集すると hash が変わるので trust が自動的に無効化されます。次回 splash でプレースホルダに戻り、再度 `splashboard trust` を通すまで動きません。

一度信頼したから、後で何が書き換わっても動き続ける、という事故が起きないようになっています。

### なぜディレクトリ単位ではないか

このディレクトリは信頼する、このレポは信頼する、という粗い粒度にはしていません。理由は、

- 信頼の単位はこのフォルダではなく、この設定の中身だから
- レポをクローンしたあとに `dashboard.toml` が悪意ある内容に書き換わるパターンを catch したいから

clone した時点での `dashboard.toml` を一度信頼しても、後から書き換えられたら hash が変わって trust は失効します。これがディレクトリ信頼との一番の違いです。

## `SPLASHBOARD_TRUST_ALL`

CI / 自動化された環境では、対話プロンプトが走らせられないことがあります。その場合の脱出口がこれです。

```bash
export SPLASHBOARD_TRUST_ALL=1
```

設定すると、全 config の trust ゲートを無視します。

ただし、これを普段のシェルに置くのは設計上の意図に反します。 trust モデルの目的はクローンした他人のレポの Network widget を自動実行しないことなので、これを常時 set すると per-repo dashboard の意義の半分が消えます。 CI や自動デプロイの中だけに留めるのが想定された使い方です。

## Exec クラスは存在しない

trust モデルには本来 3 クラス目があってもおかしくありません。サブプロセスを起動する widget ( `command = "..."` の plugin 系 ) に対する **`Exec`** クラスです。

splashboard はこのクラスを意図的に閉じています。サブプロセス widget も、 plugin 機構も存在しません。理由はシンプルで、

- 任意のコマンドを cd するだけで実行できる仕組みは、 trust モデルでは守りきれない
- カスタム widget は ReadStore か、 PR で本体に追加する形に絞る

つまり splashboard は curated な renderer であって、シェルスクリプトのホストではない、というスタンスです。拡張のしやすさとセキュリティ境界のどちらを優先するか、設計判断としては前者を諦めた、ということになります。

## secret の置き場所と漏れない仕組み

最後に secret の話です。 `github_*` や認証が要る fetcher は token を必要としますが、 token は shell rc には書かないで、専用ファイルに置く仕掛けがあります。

```toml
# $HOME/.splashboard/secrets.toml
GH_TOKEN      = "ghp_..."
TODOIST_TOKEN = "..."
```

splashboard は起動時にこのファイルを読んで、各 key をプロセスの環境変数として export します。 shell rc を編集しなくても fetcher が `GH_TOKEN` を見ます。

仕様で押さえておくとよい点が 3 つあります。

- **シェル env が常に勝つ**: 既に env に `GH_TOKEN=...` がある場合は上書きしない。 ad hoc に `GH_TOKEN=xxx splashboard ...` の override が効く
- **危険な key は無視される**: `PATH` 、 `HOME` 、 `SHELL` 、 `LD_PRELOAD` 、 `DYLD_*` 、 `SPLASHBOARD_*` などはこのファイルから export されない。 secret 経由で ambient state を書き換える攻撃を防ぐため
- **`splashboard install` が `.gitignore` に追加**: dotfiles レポに `.splashboard/` をコミットしていても、 `secrets.toml` だけは git 管理外になる

dotfiles を公開している人は、

- `home.dashboard.toml` / `project.dashboard.toml` / `settings.toml` → コミット
- `secrets.toml` / `cache/` / `trust.toml` → コミットしない

の使い分けが標準的です。 `splashboard install` が出力する `.gitignore` をそのまま使えば、何もせずに正しい分離ができます。
